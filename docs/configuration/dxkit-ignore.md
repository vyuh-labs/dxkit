# `.dxkit-ignore`

A per-project exclusion file. Same syntax as `.gitignore`. Tells
dxkit to skip files / directories from every analysis (source-file
counts, secret scans, line-counters, lint, duplicate detection,
test-gap discovery ‚Äî all of them).

## When to use it

- Vendored code you didn't write (third-party libs checked in)
- Generated files that aren't autogen-header-marked (dxkit auto-skips
  files with `<auto-generated>` / `@generated` / `DO NOT EDIT` in
  the first 2KB, but not all generators emit those)
- Demo / sample code that's not your team's responsibility
- Long-running migration artifacts you'll clean up later but shouldn't
  count against the current health score

## When NOT to use it

`.dxkit-ignore` is **path-based scoping** ‚Äî files matched here are
never scanned, no findings emitted, no scoring impact. That's the
right tool for "this code isn't ours to maintain."

It is NOT the tool for **per-finding suppression**. If dxkit scans
a file and emits a specific finding you want to suppress (false
positive, intentional test fixture, externally mitigated), use the
[allowlist](../commands/allowlist.md) ‚Äî it carries a typed category

- reason + (when relevant) expiry, and stays auditable. Ignoring
  the whole file would also silence any other finding in that file
  that you DO care about.

## File location

Place `.dxkit-ignore` at the repo root (where `.git/` lives). DXKit
auto-discovers it.

## Syntax

Standard gitignore ‚Äî same parser dxkit uses for `.gitignore`:

```gitignore
# Comments start with #
# Lines starting with ! (negation) are skipped ‚Äî dxkit treats every
# entry as additive.

# Single-segment basename: matches dir or file anywhere in the tree
generated/
old-vendor/

# Multi-segment path: anchored to the repo
Dev/Addons/SAPB1/
public/build/

# Glob: matches anywhere
*.gen.ts
*.min.js
*.bak

# Specific file
src/legacy/.old.tsx
```

## How dxkit applies it

The bundled exclusion set + project `.gitignore` + project
`.dxkit-ignore` are unioned. The canonical predicate
`isExcludedPath(cwd, relPath)` is consulted by every analyzer +
every external tool wrapper, so behavior is consistent.

Three layers, unioned:

1. **Bundled defaults** ‚Äî `node_modules`, `dist`, `build`, `vendor`,
   `target`, `bin`, `obj`, `.git`, `.cache`, `.next`, `.nuxt`, `.tox`,
   `coverage`, `.dxkit`, `__pycache__`, `.venv`, `venv`, `*.min.js`,
   `*.d.ts`, ~24 entries.
2. **Project `.gitignore`** ‚Äî your repo's existing ignore rules.
3. **Project `.dxkit-ignore`** ‚Äî your dxkit-specific extras.

You don't have to repeat `.gitignore` entries in `.dxkit-ignore` ‚Äî
they already apply.

## Verifying it took effect

After editing `.dxkit-ignore`, re-run the report. The
[`health`](../commands/health.md) summary should show fewer source
files; the prose should drop any reference to paths you excluded.

If a path still appears, double-check the glob syntax. Glob patterns
match BASENAMES anywhere in the tree by default ‚Äî `*.min.js` works.
For path-anchored exclusions, use multi-segment paths:
`public/assets/` excludes only paths matching `*/public/assets/*`.

## What dxkit will NEVER ignore

- Files inside the dxkit reports directory (`.dxkit/reports/`)
- `.git/` internals
- The `dxkit.yml` config itself

These are always read, regardless of exclusion rules ‚Äî they're meta
data, not analyzed content.
